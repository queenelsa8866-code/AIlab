一、需求背景

现状问题一：信息分散

团队需要持续追踪 AI 产品、竞品动态、研究趋势和业务场景案例，但来源分散在 RSS、官网、媒体、GitHub、社区内容和内部材料中，难以形成统一池。

现状问题二：信息不等于机会

当前更多停留在“知道了一条资讯”，但很少进一步结构化为“为什么重要”“适合哪个业务场景”“下一步谁来做什么”。

现状问题三：评审难聚焦

内部评审真正关心的是一周内出现了哪些高价值机会，而不是一长串低密度资讯列表。

为什么现在做

当前已经具备一期推荐白名单，可先用 28 个较高质量信源跑通最小闭环。产品价值不在于做一个更大的资讯站，而在于用统一规则把外部变化转换为结构化洞察和内部行动建议，降低 AI Lab 的预研与判断成本。

核心判断

外部信息 -> 标准化采集 -> 去重与摘要 -> 按统一维度筛选打分 -> 生成洞察卡 -> 汇总本周高价值机会 -> 形成建议动作与责任归属

六、详细方案

6.1 模块一 / 信息模块

页面定位

信息模块承接用户日常捕捉外部变化的主要入口。它不是单一列表，而是一组围绕“信息池、AI 二次筛选和辅助浏览”的信息产品结构。

信息结构

该模块下包含 精选、全部 AI 动态、AI 日报、潜力爆文、信源、新增信源 六个入口。其中“精选”是 AI 从信息池中二次筛出的高价值结果，“全部 AI 动态”是底层信息池。

信息模块功能说明

入口 说明
精选 展示 AI 从信息池中二次筛选后的高价值内容，优先满足“先看什么”的需求
全部 AI 动态 承接白名单采集后的全量信息池，用户可以在这里查看完整动态并回溯原始信息
AI 日报 自动汇总当天重点信息，适合快速浏览当日变化与重点趋势
潜力爆文 识别具有高传播潜力或高讨论价值的内容，帮助用户补充发现“值得多看一眼”的信息
信源 展示当前已纳入的信息来源，帮助用户理解系统覆盖边界与来源结构
新增信源 提交新的候选信息源，补充系统信息覆盖范围；该入口是信息模块里唯一带人工审核的能力

信息模块基础规则

  • 数据来源仅限一期 28 个白名单信源。
  • “全部 AI 动态”是信息池，“精选”是从信息池中经过 AI 二次筛选后的结果。
  • 新增信源需要人工审核，其余信息消费与洞察生成默认由系统直接完成。

6.2 模块二 / 需求洞察

需求洞察是产品核心。系统不是简单总结原文，而是将信息转成一张可评审的判断卡片。

洞察卡最小结构

字段 说明
事实摘要 客观概括发生了什么，避免过度加工
为什么重要 说明该变化为何值得 AI Lab 关注
适用业务场景 映射到 UU 具体业务或内部提效场景
机会建议 从产品/运营/设计/技术角度提出可讨论机会
优先级 用于初步收束，不等同于项目正式排期
置信度 表明系统对该判断的把握程度
建议动作 明确下一步谁该做什么，例如“产品本周补充业务映射”“设计补形态参考”“技术评估接入成本”

基础规则

  • 所有洞察均按统一 4 维评分口径产出，避免不同人采用不同判断标准。
  • 首版默认采用系统直出洞察，生成后直接进入结果沉淀。
  • 洞察由信息模块承接输入，但在产品结构上作为独立第二模块存在。

6.3 模块三 / 更新日志与反馈

第三模块不直接生产洞察结果，但承担系统可追踪、可修正、可持续优化的支持能力。

入口 说明
更新日志 记录信源接入、规则变化、分析策略调整和版本更新，帮助用户理解系统最近发生了什么变化
反馈 支持用户反馈筛选结果是否准确、洞察是否有帮助，以及哪些方向需要持续补充或修正

6.4 数据流与职责分工

白名单信源 -> 采集原文 / 链接 / 元信息 -> 去重与摘要 -> 全部 AI 动态（信息池） -> AI 二次筛选 -> 精选 / AI 日报 / 潜力爆文 -> 生成需求洞察 -> 洞察结果沉淀 新增信源提报 -> 人工审核 -> 纳入白名单 -> 下一轮采集生效 更新日志 / 反馈 -> 记录变化 -> 回流优化筛选与洞察规则

八、V1.0 范围

白名单信源驱动

V1.0 以一期 28 个推荐白名单源为输入基础，优先保证采集、去重、摘要与洞察链路稳定可用。

统一评分与洞察沉淀

V1.0 统一采用相关性、新颖度、可落地性、影响范围 4 个维度进行筛选，并输出结构化洞察卡。

双入口支撑评审

V1.0 在首页右侧提供“机会策略”和“新增信息源”两个侧边栏入口，分别承接高价值机会评审与信源扩充。
